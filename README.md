### 1. Описание функциональности монолитного приложения

**Авторизация в Web приложении**
- Пользователи могут войти в приложение для просмотра температуры и управления отоплением своего дома.
- Администратор системы может войти в приложение для регистрации нового датчика.

**Управление отоплением:**

- Пользователи могут управлять отоплением в доме.
- Система поддерживает удаленное управление отоплением с помощью специальных датчиков и реле через приложение компании.
- Для подключения к системе необходимо вызывать специалиста по подключению системы отопления в доме.

**Мониторинг температуры:**

- Пользователи могут проверять температуру.
- Система поддерживает мониторинг температуры через приложение уомпании удаленно.
- Датчики регистрируются при включении.

**Регистрация датчика**

- Администратор системы может зарегистрировать новый датчик.

**Управление пользователями системы**

- Администратор системы может добавить нового пользователя в систему.
- Администратор системы может удалить пользователя из системы.

### 2. Анализ архитектуры монолитного приложения

Язык программирования: Go.
СУБД: Postgres.
Взаимодействие между компонентами: синхронно, никаких асинхронных вызовов. Все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
Масштабируемость: ограничена, так как монолит сложно масштабировать по частям.
Развертывание: требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

Домены:
- Управление устройствами
- Мониторинг температуры

В домене управление отполением поддоменами могут быть:
- Управление датчиками системы отопления

В домене мониторинг температуры поддоменами могут быть:
- Анализ температуры

В поддомене управление датчиками системы отопления контекстами могут быть:
- контекст включения отопления (с терминами "включение")
- контекст выключения отопления (с терминами "выключение")
[Context.puml](Context.puml)
В поддомене анализ температуры контекстами могут быть:
- контекст просмотра текущей температуры (с терминами "единицы измерения")

### **4. Проблемы монолитного решения**

- Масштабирование системы затруднено, при увеличении нагрузки увеличивать ресурсы необходимо для всего приложения.
- Для подключения системы отопления в доме нужен выезд специалиста (не проблема монолита как архитектурного подхода, а проблема текущей системы в монолитном представлении).
- Синхронное взаимодействие между компонентами при увеличении нагрузки может полностью вывести систему из строя и повлечь за собой потерю данных пользователей.
- При внесении изменений в текущей версии необходимо проводить тестирование всей системы, из-за чего увеличивается время выпуска релиза.
- Во время выхода нового релиза вся система какой-то период времени будет недоступна пользователям.
